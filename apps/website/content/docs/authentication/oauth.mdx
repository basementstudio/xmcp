`xmcp` supports native OAuth configuration for custom OAuth/resource-server setups through the stable top-level `oauth` config.

Use this when you need to advertise OAuth metadata from your MCP server while proxying authorization and token flows to your own authorization server, or when you want to combine xmcp's OAuth routes with custom verification middleware.

```typescript title="xmcp.config.ts"
import { type XmcpConfig } from "xmcp";

const config: XmcpConfig = {
  http: true,
  oauth: {
    issuerUrl: "https://auth.example.com",
    baseUrl: "https://mcp.example.com",
    middleware: true,
  },
};

export default config;
```

By default xmcp performs issuer discovery against `/.well-known/oauth-authorization-server` and falls back to `/.well-known/openid-configuration`. Use `oauth.endpoints` only when you need to override missing or non-standard provider metadata.

When the built-in middleware validates a token, your tools can read the normalized auth context through `xmcp/auth`.

If you want to keep the OAuth metadata/routes but use your own request verification, set `oauth.middleware` to `false` and provide custom validation in `src/middleware.ts`.

If you are migrating from `experimental.oauth`, move the same object to `oauth`. The compatibility alias still works for now, but the stable key takes precedence when both are defined.

When both JWKS and introspection are configured, xmcp prefers local JWKS validation first. That keeps request handling fast, but it also means revoked JWTs remain valid until expiry unless you rely on introspection or custom middleware.

## Runnable examples

- Built-in native OAuth middleware: run `pnpm --dir examples/auth-native-oauth dev`
- Custom middleware with native OAuth routes: run `pnpm --dir examples/auth-native-oauth-custom-middleware dev`
