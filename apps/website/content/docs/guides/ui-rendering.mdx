## Overview

`@xmcp-dev/ui` supports two related use cases:

- schema-driven UI rendering
- MCP App runtime helpers for handwritten React apps

It provides four main entry points:

- `createRenderJsonTool()` for the normal schema-driven path
- `Rendered` for custom tool wrappers with the built-in preview engine
- `App` for full control over the renderer
- `useMcpApp()` for handwritten React MCP Apps that talk to the host directly

For most projects, start with `createRenderJsonTool()`. Use the other entry
points only when you need more control.

## Quickstart: packaged `renderJson`

Import the package stylesheet once:

```css title="src/app/globals.css"
@import "tailwindcss";
@import "@xmcp-dev/ui/styles.css";

@theme {
  --font-sans: "Geist", ui-sans-serif, sans-serif;
}
```

Expose a tiny local `render-json` tool:

```tsx title="src/tools/render-json.tsx"
import { createRenderJsonTool } from "@xmcp-dev/ui";

const renderJsonTool = createRenderJsonTool();

export const metadata = renderJsonTool.metadata;
export const schema = renderJsonTool.schema;
export default renderJsonTool.handler;
```

The normal tool input stays simple:

```json
{
  "schemaJson": "{ \"title\": \"Example App\", \"mcpServerUrl\": \"https://example.com\", \"root\": { \"type\": \"grid\", \"props\": { \"columns\": 1, \"gap\": 16 }, \"children\": [] } }"
}
```

The package handles the normal schema-driven rendering flow for you.

## Host-backed schema apps

If the same schema-driven app should route tool calls and actions through the
MCP App host, use host-backed transport:

```tsx title="src/tools/render-json.tsx"
import { createRenderJsonTool } from "@xmcp-dev/ui";

const renderJsonTool = createRenderJsonTool({
  transportMode: "host",
});

export const metadata = renderJsonTool.metadata;
export const schema = renderJsonTool.schema;
export default renderJsonTool.handler;
```

You can also use `transportMode="auto"` to prefer the host when connected and
fall back to HTTP outside supported hosts.

The same transport option exists on `Rendered` and `App`.

## When should you use each path?

**Use `createRenderJsonTool()`** when you want the normal schema-driven MCP
server path. This is the default and the easiest place to start.

**Use `Rendered`** when you want a custom tool wrapper but still want the
built-in preview engine.

**Use `App`** when you want to own parsing, validation, loading states, and the
renderer pipeline directly.

**Use `useMcpApp()`** when you are building a handwritten React MCP App that
should talk to the host directly.

<Callout variant="info">
  Start with `createRenderJsonTool()`. Move to `Rendered` only when you need a
  custom wrapper, move to `App` when you want direct renderer control, and use
  `useMcpApp()` for handwritten React MCP Apps.
</Callout>

## What the packaged path does for you

The packaged `renderJson` path renders through `Rendered`. It keeps the common
case simple and handles common model-generated mistakes.

By default, it uses progressive previewing, light mode, and the `zinc` preset.
While the model is still streaming, it can attempt partial JSON recovery
instead of waiting for the full payload.

It also normalizes common safe sizing mistakes and rejects unreadable inline
`themeTokens`, falling back to the packaged preset theme when needed.

This lets the server author control defaults while the model usually only needs
to supply `schemaJson`.

## Custom React MCP Apps with `useMcpApp()`

For handwritten React UIs, `useMcpApp()` is the recommended hook:

```tsx title="src/tools/ops-console.tsx"
import { Button, useMcpApp } from "@xmcp-dev/ui";

export default function OpsConsole() {
  const { callTool, requestDisplayMode, isConnected } = useMcpApp();

  return (
    <Button
      onClick={async () => {
        if (isConnected) {
          await requestDisplayMode("fullscreen");
        }

        await callTool("serverStats");
      }}
    >
      Run tool
    </Button>
  );
}
```

`useMcpApp()` exposes MCP App capabilities such as:

- `callTool`
- `openLink`
- `requestDisplayMode`
- `readResource`
- `sendMessage`
- `updateModelContext`
- `hostContext`
- `hostCapabilities`

`useMcpHostBridge()` still exists as a compatibility alias, but `useMcpApp()`
is the preferred hook name.

## Auto size reporting

If your host supports size notifications, `useAutoMcpAppSize()` uses
`ResizeObserver` to report `ui/notifications/size-changed` from your app root.

## Using `Rendered`

Use this when you want your own tool definition and still want the built-in
preview engine.

```tsx title="src/tools/render-dashboard.tsx"
import { Rendered } from "@xmcp-dev/ui";
import { z } from "zod";

export const schema = {
  schemaJson: z.string().optional(),
};

export const metadata = {
  name: "renderDashboard",
  description: "Preview an AppSchema with server-owned defaults.",
};

export default function renderDashboard({
  schemaJson,
}: {
  schemaJson?: string;
}) {
  return (
    <Rendered
      schemaJson={schemaJson}
      previewMode="progressive"
      themeMode="light"
      themePreset="zinc"
      transportMode="auto"
    />
  );
}
```

## Using `App` directly

Use this when you want to own the full preview pipeline.

```tsx title="src/tools/render-strict-app.tsx"
import { App, validateSchema } from "@xmcp-dev/ui";
import { z } from "zod";

export const schema = {
  schemaJson: z.string().optional(),
};

export const metadata = {
  name: "renderStrictApp",
  description: "Render a validated AppSchema with custom server-side behavior.",
};

export default function renderStrictApp({
  schemaJson,
}: {
  schemaJson?: string;
}) {
  if (!schemaJson?.trim()) {
    return <div>Waiting for schema JSON...</div>;
  }

  try {
    const parsed = JSON.parse(schemaJson);
    const appSchema = validateSchema(parsed);
    return <App schema={appSchema} transportMode="auto" />;
  } catch (error) {
    return <pre>{error instanceof Error ? error.message : String(error)}</pre>;
  }
}
```

## Framework-agnostic MCP App runtime

If you are not using `@xmcp-dev/ui`, the host runtime is available from the main
`xmcp` package:

```ts
import { createMcpHostBridge } from "xmcp/host-bridge";
```

This is the public import path for the MCP App host bridge.
