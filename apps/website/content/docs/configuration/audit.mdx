`xmcp audit` runs a static analysis pass over your MCP server and surfaces security, compliance, quality, and performance issues — things like hardcoded secrets, `z.any()` in tool schemas, confused-deputy token passthrough, or a tool description that drifted from its schema.

## Commands

```bash
xmcp audit                      # scan the project
xmcp audit:list-rules           # list every rule, grouped by concern
xmcp audit:explain <rule-id>    # rationale + bad/good examples for one rule
xmcp build --audit              # run deterministic audit checks before bundling
```

Common flags:

| Flag                             | Purpose                                                                                                                |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `--format terminal\|json\|sarif` | Output format — SARIF integrates with GitHub code scanning.                                                            |
| `--severity <level>`             | Minimum severity to show (`info`, `low`, `medium`, `high`, `critical`).                                                |
| `--fail-on <level>`              | Exit 1 if any finding is at or above this level (default `high`).                                                      |
| `--no-heuristics`                | Skip heuristic rules for deterministic automation and build preflights.                                                |
| `--baseline [file]`              | Only fail on findings not already in the baseline. Pair with `--update-baseline` to write one.                         |
| `--since <ref>` / `--changed`    | Only scan files changed since a git ref / in the working tree. Project-wide rules still run against the whole project. |
| `--ci`                           | No colors; implies `--fail-on high`.                                                                                   |

## Configuration

Every audit setting is optional. Omit the `audit` block and you get the defaults.

```typescript title="xmcp.config.ts"
const config = {
  audit: {
    ignore: [
      "src/generated/**",
      { rule: "XMCP-PERF-002", paths: ["src/tools/legacy/**"] },
    ],
    severity: {
      "XMCP-SCHEMA-004": "off",
      "XMCP-META-001": "high",
    },
    failOn: "high",
  },
};

export default config;
```

- **`ignore`** — glob string (matches every rule on that path) or `{ rule, paths }` (matches only the given rule on the given paths). POSIX separators, relative to the project root.
- **`severity`** — remap a rule to a different level, or `"off"` to disable it entirely.
- **`failOn`** — default threshold for a non-zero exit code. Overridden by CLI `--fail-on`.

For one-off suppressions, annotate the line instead:

```typescript
// xmcp-audit-ignore-next-line XMCP-SCHEMA-001 payload is truly opaque here
payload: z.any(),
```

## Build preflight

Use `xmcp build --audit` to run the deterministic audit subset before bundling. This preflight skips dependency audit and heuristic rules, then aborts the build if findings meet the configured threshold.

Use it when you want framework users to catch obvious security, schema, and config issues locally before the compiler starts work.

## CI/CD

`xmcp audit` is designed to fit downstream MCP project pipelines.

For a blocking PR gate, prefer deterministic static checks only:

```bash
xmcp audit --no-heuristics --no-deps --fail-on high --format json
```

For scheduled security scans or code-scanning integrations, prefer SARIF and keep dependency audit enabled:

```bash
xmcp audit --format sarif --output audit.sarif
```

Recommended rollout:

- Use `--no-heuristics --no-deps` for fast, predictable blocking checks.
- Use SARIF for scheduled scans or security dashboards.
- Add baselines only when an existing codebase needs staged adoption.

## Rules

The full rule catalog — IDs, severities, rationale, and detection logic — lives in the repo:

[packages/xmcp/src/cli/commands/audit/rules](https://github.com/basementstudio/xmcp/tree/main/packages/xmcp/src/cli/commands/audit/rules)

Rules are grouped by concern: `security`, `compliance`, `quality`, `performance`. Run `xmcp audit:list-rules` for a local listing, or `xmcp audit:explain <rule-id>` to see the rationale and examples for any single rule.
